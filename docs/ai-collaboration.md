# AgentHub AI 协作开发记录

## 协作目标

本项目将 AI 协作能力作为核心评分点进行设计。目标不是只让 AI 生成代码，而是沉淀一套可复用的 Spec、Rules、Skills 和开发日志，使仓库本身对 Agent 友好，后续任何 AI Agent 都能快速理解项目边界并继续开发。

## 协作分工

| 角色 | 职责 | 产出 |
| --- | --- | --- |
| Human PM | 明确比赛目标、评估取舍、决定优先级 | 题面理解、验收标准 |
| Orchestrator | 拆解复杂任务、分派 Agent、聚合结果 | 任务计划、进度同步 |
| Designer Agent | 设计 IM 布局和演示路径 | 三栏 UI、交互细节 |
| Codex Agent | 实现静态 Demo、消息逻辑和卡片渲染 | `app/` 代码 |
| Tech Writer Agent | 整理产品和技术文档 | `docs/` 文档 |
| QA Agent | 检查可运行性、中文完整性和交付完整度 | 验证记录 |

## Spec

### 竞赛验收 Spec

- 必须提供产品设计文档。
- 必须提供技术设计文档。
- 必须提供可运行 Demo。
- 必须提供 AI 协作开发记录。
- Demo 需要体现 IM 聊天、多 Agent 调度、产物预览和部署发布。
- 真实 API Key 不得硬编码到前端或提交到公开仓库。

### Demo 功能 Spec

- 用户能看到会话列表和 Agent 联系人。
- 用户能创建单聊或群聊会话。
- 用户能发送消息并触发多 Agent 回复。
- 消息流能展示文本、代码、Diff、预览和部署卡片。
- 用户能 pin 上下文、引用消息、复制内容、重新生成回复。
- 用户能点击应用 Diff，并看到右侧预览和版本历史更新。
- 用户能点击部署，并看到部署状态卡片。

## Rules

### 代码规则

- 优先做可演示闭环，不引入不必要依赖。
- Demo 版本使用原生 HTML/CSS/JS，降低运行门槛。
- 真实接入能力通过 Adapter 抽象预留，不在前端暴露密钥。
- 高风险操作，例如部署、写文件、删除文件，在真实版本中必须二次确认。

### 产品规则

- 所有 Agent 都以“联系人”方式出现，符合 IM 心智。
- 复杂任务默认进入群聊，由 Orchestrator 拆解。
- 产物必须内联在消息流中，而不是跳转到外部页面后丢失上下文。
- 用户能随时从“聊天过程”定位到“产物结果”。

### 文档规则

- 先写可验收内容，再写愿景扩展。
- 不夸大 Demo 能力，明确区分“已实现模拟”和“真实版本规划”。
- 涉及密钥只写变量名，不写真实值。

## Skills

### Orchestrator Skill

```text
当用户输入复杂目标时：
1. 总结用户目标。
2. 拆成产品、设计、代码、部署、验证子任务。
3. 为每个子任务选择 Agent。
4. 要求 Agent 输出结构化消息和可渲染 artifact。
5. 汇总结果并给出下一步建议。
```

### Code Agent Skill

```text
当需要实现功能时：
1. 先确认影响文件。
2. 优先最小可用实现。
3. 输出代码块或 Diff artifact。
4. 说明如何验证。
5. 避免修改无关逻辑。
```

### QA Skill

```text
当准备交付时：
1. 检查 README 是否能指导运行。
2. 检查 Demo 是否能打开。
3. 检查产品、技术、AI 协作文档是否齐全。
4. 检查中文是否乱码。
5. 检查仓库是否包含明文密钥。
```
